# Harness Engineering — 核心概念

测试装具（Test Harness）是一套让你能够**自动、重复、可靠**地运行测试的基础设施。

---

## 1. 为什么需要 Harness？

手动测试的问题：
- 每次改动都要重新点击 → **慢**
- 人的注意力有限 → **不可靠**
- 环境不同结果不同 → **不可复现**

Harness 解决了以上三点：代码驱动、确定性结果、任何机器都能跑。

---

## 2. 核心组件

```
┌────────────────────────────────────────────────────────┐
│                     Test Harness                       │
│                                                        │
│  ┌──────────┐   调用    ┌──────────┐   验证  ┌──────┐  │
│  │  Driver  │ ────────▶ │   SUT    │ ──────▶ │Assert│  │
│  │（驱动器）│          │（被测系统）│         │（断言）│  │
│  └──────────┘          └──────────┘         └──────┘  │
│        │                    │                          │
│        ▼                    ▼                          │
│  ┌──────────┐         ┌──────────┐                     │
│  │ Fixture  │         │  Stub/   │                     │
│  │（测试夹具）│         │  Mock    │                     │
│  └──────────┘         └──────────┘                     │
└────────────────────────────────────────────────────────┘
```

- **SUT（System Under Test）**：被测的函数、类或服务
- **Driver**：调用 SUT 的测试代码
- **Fixture**：为测试准备的固定状态（数据库记录、文件、配置）
- **Stub/Mock**：替代真实依赖的假实现

## 3. 五种测试替身（Test Double）

Gerard Meszaros 在 *xUnit Test Patterns* 中定义了五种：

### Dummy（哑对象）
```
用途：填充参数，从不被实际调用
特点：可以是 null 或任意占位值
场景：函数需要 5 个参数但你只关心其中 2 个
```

### Stub（桩）
```
用途：返回预设值
特点：不验证谁调用了它，只负责返回
场景：替代数据库查询，让它返回固定数据
```

### Spy（间谍）
```
用途：记录调用信息，同时保留真实行为
特点：是真实对象的包装，额外记录被调用情况
场景：验证某方法被调用了几次、传了什么参数
```

### Mock（模拟对象）
```
用途：预先声明期望，事后自动验证
特点：主动的——如果期望不满足，测试失败
场景：验证第三方 API 被正确调用
```

### Fake（伪对象）
```
用途：真实的轻量替代品
特点：有真实的业务逻辑，但简化了实现
场景：SQLite 替代 PostgreSQL，内存 Map 替代 Redis
```

> **记忆口诀**：Dummy 占位，Stub 给值，Spy 偷看，Mock 验证，Fake 真做。

## 4. 测试金字塔

```
        ╱‾‾‾‾‾‾‾‾‾‾‾‾‾╲
       ╱   E2E Tests    ╲        少量 · 慢 · 贵
      ╱─────────────────╲
     ╱  Integration Tests ╲      适量 · 中等速度
    ╱───────────────────────╲
   ╱      Unit Tests         ╲   大量 · 快 · 便宜
  ╱─────────────────────────╲
```

**比例参考**：Unit 70% · Integration 20% · E2E 10%

| 层级 | 测试对象 | 速度 | 依赖隔离 | 维护成本 |
|------|---------|------|---------|---------|
| Unit | 单函数/类 | 毫秒 | 完全隔离 | 低 |
| Integration | 模块协作 | 秒 | 部分真实 | 中 |
| E2E | 用户场景 | 分钟 | 无隔离 | 高 |

## 5. 测试生命周期（AAA 模式）

每个测试都应遵循 **Arrange → Act → Assert**：

```
┌─────────────────────────────────────────┐
│  Setup (beforeAll / beforeEach)          │  ← 整个套件/每个测试前执行
├─────────────────────────────────────────┤
│  ARRANGE  准备测试环境、数据、Mock         │
│     ↓                                   │
│  ACT      调用被测代码                    │
│     ↓                                   │
│  ASSERT   验证结果是否符合预期             │
├─────────────────────────────────────────┤
│  Teardown (afterAll / afterEach)         │  ← 清理资源、回滚数据
└─────────────────────────────────────────┘
```

## 6. FIRST 原则

好的测试必须满足：

| 字母 | 含义 | 具体要求 |
|------|------|---------|
| **F**ast | 快速 | 单元测试 < 100ms，全套 < 5min |
| **I**ndependent | 独立 | 测试 A 不依赖测试 B 的结果 |
| **R**epeatable | 可重复 | 不同机器、不同时间，结果相同 |
| **S**elf-validating | 自验证 | 只有通过/失败，不需要人判断 |
| **T**imely | 及时 | 写代码时同步写测试，不是事后补 |

## 7. 覆盖率不等于质量

```python
# 100% 覆盖率，但测试毫无意义
def test_add():
    result = add(1, 2)
    assert result is not None   # 只验证了"不是 None"
```

覆盖率是**最低门槛**，不是质量保证。真正有价值的是：
- 覆盖了**边界条件**（0、-1、最大值、空字符串）
- 覆盖了**异常路径**（网络超时、数据库错误）
- 断言了**具体的业务行为**，而非"能运行"
